# Response: IndicDB CSV-to-Relational Pipeline (Project-Specific)

### 1. The Normalization Logic
- **Entity Extraction:** We split raw wide CSVs into thematic entities using a schema spec generated by an LLM and then formalized in `schema_config.yaml`. In practice:
  - stable descriptor attributes (geography, time, metadata categories) are mapped to `DIM_*` tables,
  - measurable/analytic attributes are mapped to `FACT_*` tables,
  - repeated high-redundancy attributes are deduplicated using `dedup_columns` in each dimension block.
- **Main vs Lookup/Dimension decision rule:** If a field is used to define context (who/where/when), it is usually placed in dimension tables; if it is a quantity/outcome/metric, it is placed in fact tables.
- **Schema Type:** Most datasets are **star/snowflake-style normalized relational schemas**. Across created DBs, most contain both `DIM_*` and `FACT_*` tables, with explicit inter-table links.

### 2. The Linking Strategy (Primary & Foreign Keys)
- **Relationship Discovery:** Primarily **metadata-driven (Option C)** using the parsed schema plan:
  - `parse_schema_to_yaml.py` converts `schema_info.md` into `schema_config.yaml`,
  - this YAML explicitly defines `key_column` and `foreign_keys`,
  - `generate_ddl.py` materializes these FK declarations into `DDL.csv`,
  - `create_tables.py` executes DDL, creating relational constraints in PostgreSQL.
- **Name matching/value overlap:** Not the primary FK discovery mechanism; keys are set by schema design + config.
- **Composite Keys:** Where uniqueness is hierarchical (e.g., geography levels), the pipeline generally introduces a surrogate `*_ID` key and keeps hierarchy columns as attributes. Composite semantic identifiers can still exist in source columns, but FK links are typically implemented via surrogate keys.

### 4. The Pipeline Workflow (Step-by-Step)
1. **Ingestion:** Place raw CSV as `data/total_data.csv` under a dataset folder.
2. **Schema Design:** `create_schema_llm_judge.py` proposes normalized DIM/FACT structure and link plan.
3. **Schema Parsing:** `parse_schema_to_yaml.py` converts markdown design to machine-readable `schema_config.yaml`.
4. **Normalization/Splitting:** `generic_split.py` uses YAML mappings to generate per-table CSVs and sample JSONs.
5. **DDL Generation:** `generate_ddl.py` infers data types and emits FK-aware `DDL.csv`.
6. **Table Creation:** `create_tables.py` creates schema + tables in PostgreSQL.
7. **Data Loading:** `load_data.py` bulk-loads table CSV files into corresponding tables.
8. **Validation/Querying:** `run_query.py` and query validation scripts are used for integrity and execution checks.

### 5. Handling "Dirty" Data (The Real-World Challenge)
- **Human-in-the-loop:** The process is **not fully automatic** end-to-end.
  - LLM schema proposals and glossary-based mappings are reviewed/iterated by the developer.
  - FK structure and naming are corrected where needed before final load.
- **Examples of cleaning/normalization pressure handled by the pipeline:**
  - cryptic source headers mapped into explanatory names through schema/glossary interpretation,
  - deduplication of repeated categorical combinations into dimensions,
  - type normalization for IDs, flags, year-like fields, financial/measurement fields.

### 6. Visuals & Graphs (Architecture Diagram Blocks)
- **Block A: Data Acquisition & Profiling**
  - NDAP/IDP/IHDS CSV ingestion, header/glossary inspection.
- **Block B: Semantic Schema Synthesis**
  - LLM Architect/Auditor/Refiner → `schema_info.md` → `schema_config.yaml`.
- **Block C: Relational Materialization**
  - split engine + DDL generation + FK creation (`generic_split.py`, `generate_ddl.py`, `create_tables.py`).
- **Block D: Warehouse Loading & Benchmark Readiness**
  - PostgreSQL loading (`load_data.py`), schema-level validation, query execution/evaluation setup.

### Add-on Complexity Signals You Can Cite
- Average tables/database is high (11.85 in created design set), indicating multi-table reasoning pressure.
- Join graphs include deep chains (up to depth ~6 in current schemas), not only flat star joins.
- Live deployment scale is multi-million rows with high-cardinality geography entities, making value grounding non-trivial.

## Agentic Approach and Prompts Used (Implemented Pipeline)

Source: `scripts/create_schema_llm_judge.py`

### Agentic Workflow (3-Agent Judge Pattern)
1. **Architect Agent**
- Input: Indexed CSV columns (`[0] col_name ...`).
- Goal: Draft a normalized multi-table schema with domain isolation.
- Output: Initial DIM/FACT table plan.

2. **Auditor Agent**
- Input: Architect draft.
- Goal: Detect table clumping, width violations, cohesion/3NF issues, and low join complexity.
- Output: Explicit failure report for refinement.

3. **Refiner Agent**
- Input: Architect draft + Auditor feedback + indexed CSV columns.
- Goal: Produce final parser-friendly schema with strict format and column mapping.
- Output: Final `schema_info.md` used by downstream parser.

### Prompt 1: Architect (Exact Intent)
```text
Role: Senior Database Architect.
Task: Analyze CSV columns with indices.
CRITICAL PRIORITY: Domain Isolation.
- Divide into 4-10 thematic tables.
- No table >12-15 columns.
- Every table has PK.
- Link tables via FKs.
- Total mapped columns target 40-80 (can drop full categories).
- Naming: UPPERCASE_WITH_UNDERSCORES.
- DIM_ prefix for dimensions, FACT_ prefix for facts.
- Do not include indexing recommendations.
```

### Prompt 2: Auditor (Exact Intent)
```text
Role: Database Normalization & Domain Auditor.
Audit checks:
1) Width Check: any table >15 columns -> God Table.
2) Cohesion Check: off-theme columns in table.
3) 3NF Violation: transitive dependencies.
4) Complexity Check: complex questions should require >=3 table joins.
Output: specific clumping and normalization failures.
```

### Prompt 3: Refiner (Exact Intent + Output Contract)
```text
Role: Lead Architect.
Task:
- Resolve auditor warnings.
- Split wide tables into thematic sub-domains.
- Ensure clear join paths.
- Clean noisy header text (e.g., units/scaling strings).

CRITICAL OUTPUT FORMAT:
1) "## DIMENSION TABLES"
   - ### **TABLE_NAME**
   - **Purpose**
   - **Columns (N):**
     - `COL` (PK/FK/type)
2) "## FACT TABLES" (same format)
3) "## COLUMN MAPPING" markdown table with:
   | Original Column | Source Index | Data Type | Mapped To Table | Mapped Column(s) | Notes |

Hard constraint:
- Use exact original CSV column names in mapping.
- Use exact 0-based source indices.
- One-to-one mapping only (no unpivoting).
- No index recommendations.
```

### Execution Mode
- **Anthropic backend:** sequential Architect -> Auditor -> Refiner calls through LangChain.
- **OpenRouter backend:** same 3-step flow; optional `--reasoning` keeps cross-step conversation history (`invoke_with_history`) so later agents inherit prior reasoning context.

### Pipeline Artifacts Produced
- `schema_info.md` (agentic output)
- `schema_config.yaml` (parsed machine-readable config)
- split table files + `DDL.csv` + loaded PostgreSQL schema.
